# 新增文档页

本站的顶部导航与侧边栏由 `docs/.vitepress/config.mts` 在构建/启动时**自动扫描 `docs/` 目录生成**。新增一篇文档**只需创建一个 `.md` 文件**并写好一级标题——菜单会自动出现，**不要手动改 `config.mts`**。

## 开始前先确认这三项

如果用户没说清，先问（一次问全，不要逐条挤牙膏）：

1. **文档标题**：显示在页面顶部、同时作为侧边栏菜单名的中文标题，例如「部署指南」。
2. **所属分组**：对应 `docs/` 下的子目录，例如 `guide`。已有目录见下方「读取现状」。
3. **文件名 slug**：小写 kebab-case 英文，例如 `deploy`，最终 URL 为 `/<分组>/<slug>`。

## 读取现状

查看 `docs/` 下已有哪些子目录（即已有分组），确认是放进现有分组还是新建分组目录。

## 创建 Markdown 文件

在 `docs/<分组>/<slug>.md` 新建文件。模板（遵循 CLAUDE.md 的「文档写作规范」）：

```markdown
# <文档标题>

<一句话说明本页主题。>

## <小节标题>

正文……
```

要点：

- 以**单个** `#` 一级标题开头，它既是页面标题，也是**自动生成的侧边栏菜单名**。
- 中文撰写，代码块标注语言。
- 若需新建分组目录（例如第一次建 `docs/notes/`），且希望分组名是中文，**额外在该目录建一个 `index.md` 并写 `# 分组名`**；否则分组名会回退成英文目录名。
- 控制排序：在 frontmatter 写 `order: <数字>`（小在前），或用 `01-`、`02-` 文件名前缀。

## 验证

```bash
npm run docs:build
```

`ignoreDeadLinks: false`，任何死链都会让构建失败；构建通过即说明文件与自动生成的菜单链接自洽。

::: tip 本地 dev 提示
若正在跑 `npm run docs:dev`，新建文件后菜单不会即时刷新，需**重启 dev server**；`docs:build` 与线上部署每次全新扫描，始终准确。
:::

## 收尾

提示用户发布：`git add . && git commit -m "docs: 新增<标题>文档" && git push`（push 到 `main` 自动部署）。commit 信息遵循 CLAUDE.md 的 Conventional Commits 规范。
